# Статический сайт медицинской организации (React + Vite)

Одностраничный информационный сайт без сервера, базы данных и админки. Контент задаётся в одном файле данных, оформление — в компонентах и стилях.

## 1. Как запустить локально

Требуется установленный [Node.js](https://nodejs.org/) (LTS, например 20.x).

```bash
npm install
npm run dev
```

Откройте в браузере адрес, который покажет Vite (обычно `http://localhost:5173/`).

Остановка сервера разработки: `Ctrl+C` в терминале.

Просмотр продакшен-сборки локально:

```bash
npm run build
npm run preview
```

## 2. Как заменить тексты и данные

Все тексты, списки услуг, контакты, карточки сотрудников и подписи разделов находятся в файле:

**`src/data/siteData.js`**

Отредактируйте нужные поля и сохраните файл. Заголовок страницы (`document.title`) берётся из `meta.siteTitle`. Ссылки якорного меню соответствуют полю `navigation` и атрибутам `id` секций в `src/App.jsx` — при переименовании раздела проверьте совпадение `id` и пунктов меню.

Путь к PDF лицензии задаётся в `siteData.license.pdfPath` (по умолчанию `docs/license.pdf` относительно каталога `public`).

## 3. Как заменить файл лицензии

Подготовьте свой PDF и положите его по пути:

**`public/docs/license.pdf`**

Имя файла должно остаться тем же (`license.pdf`), если вы не меняете `pdfPath` в `siteData.js`. При замене файла достаточно пересобрать и снова задеплоить сайт — ссылка «Скачать лицензию» автоматически указывает на новый файл, так как он копируется в сборку без изменения адреса в коде.

Проверка после сборки: в каталоге `dist/docs/` должен появиться актуальный `license.pdf`.

## 4. Как задеплоить на GitHub Pages

### Вариант A — относительный базовый путь (рекомендуется для репозитория-проекта)

В **`vite.config.js`** по умолчанию используется `base: './'` (через переменную окружения `VITE_BASE`). Так сборка корректно открывается по адресу вида `https://<пользователь>.github.io/<репозиторий>/` без ручной подстановки имени репозитория в конфиг.

```bash
npm run build
```

Загрузите содержимое папки **`dist/`** в ветку, из которой GitHub Pages отдаёт статику (часто `gh-pages` или каталог `/docs` в `main` — см. настройки репозитория **Settings → Pages**).

Простой способ без отдельных инструментов:

1. Создайте репозиторий на GitHub и загрузите проект (включая `public/docs/license.pdf`).
2. В **Settings → Pages** выберите источник **GitHub Actions**.
3. Добавьте workflow (или используйте «Upload artifact» вручную): на каждый push в `main` выполняйте `npm ci`, `npm run build`, публикуйте папку `dist` как страницу (официальный шаблон «Static HTML» или generic «Deploy static site»).

Если используете публикацию из ветки **`gh-pages`**, можно один раз выполнить из корня проекта после сборки:

```bash
npm run build
cd dist && git init && git add -A && git commit -m "Deploy" \
  && git branch -M gh-pages \
  && git remote add origin https://github.com/<USER>/<REPO>.git \
  && git push -u origin gh-pages --force
```

(Подставьте свой URL репозитория; способ требует прав на запись.)

### Вариант B — фиксированный префикс `/имя-репозитория/`

Если хотите явно задать базовый URL:

```bash
VITE_BASE=/имя-репозитория/ npm run build
```

Убедитесь, что значение совпадает с именем репозитория GitHub Pages.

### Замечания

- Файлы из **`public/`** попадают в корень **`dist/`**, поэтому PDF доступен как `…/docs/license.pdf` после деплоя.
- Репозиторий может быть общедоступным — GitHub Pages для публичных репозиториев бесплатен в рамках текущих правил GitHub.
